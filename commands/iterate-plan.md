# Iterate Plan (계획서 수정)

기존 구현 계획서를 사용자 피드백 기반으로 업데이트합니다.
코드베이스 리서치로 변경의 기술적 타당성을 검증한 후 수정합니다.

---

## 실행 시작

### 계획서 + 피드백이 모두 주어졌을 때
- 즉시 Step 1로 진행

### 계획서만 주어졌을 때
```
계획서를 확인했습니다: [경로]

어떤 변경이 필요한가요?

예시:
- "Phase 2에 마이그레이션 처리 단계 추가"
- "검증 기준에 성능 테스트 포함"
- "범위에서 기능 X 제외"
- "Phase 2를 백엔드/프론트엔드로 분리"
```

### 파라미터 없이 실행했을 때

1. `.plans/` 디렉토리에서 최근 계획서 탐색
2. 파일이 없으면 경로 요청
3. 파일이 여러 개면 목록 제시

```
수정할 계획서를 알려주세요.

최근 계획서:
- .plans/2026-04-08-add-search-filter.md
- .plans/2026-04-05-auth-refactor.md

팁: /iterate-plan .plans/파일명.md 변경사항 설명
```

---

## Step 1: 현재 계획서 이해

1. **계획서 전체 읽기** (limit/offset 없이)
   - 현재 구조, Phase, 범위 파악
   - 검증 기준, 구현 접근 방식 이해

2. **요청된 변경 분석**
   - 추가/수정/제거할 내용 파악
   - 코드베이스 리서치가 필요한지 판단
   - 변경 범위 결정

---

## Step 2: 리서치 (필요한 경우만)

**변경이 새로운 기술적 이해를 필요로 할 때만** 리서치 수행.

1. **병렬 에이전트 실행** (변경에 필요한 것만 선택):

   | 상황 | 에이전트 |
   |------|---------|
   | 새 코드 영역 관련 변경 | `codebase-locator` — 파일 위치 탐색 |
   | 구현 상세 확인 필요 | `codebase-analyzer` — 구현 분석 |
   | 유사 패턴 참고 필요 | `codebase-pattern-finder` — 패턴 탐색 |
   | 과거 관련 문서 확인 | `docs-locator` — 문서 탐색 |
   | 과거 문서 분석 필요 | `docs-analyzer` — 인사이트 추출 |

2. **리서치 결과로 발견된 파일 직접 읽기**
3. **모든 에이전트 완료 대기** 후 종합

---

## Step 3: 변경 방향 확인

수정 전에 이해한 내용 확인:

```
피드백을 분석했습니다:

**요청된 변경:**
- [변경 1 구체적 설명]
- [변경 2 구체적 설명]

**리서치 결과:** (해당 시)
- [관련 코드 패턴이나 제약]
- [변경에 영향을 주는 발견]

**수정 계획:**
1. [구체적 수정 항목]
2. [구체적 수정 항목]

이 방향이 맞나요?
```

사용자 확인 후 진행.

---

## Step 4: 계획서 수정

1. **정밀한 편집** (Edit 도구로 수술적 변경)
   - 기존 구조 유지 (명시적 변경 요청 아닌 한)
   - file:line 참조 정확하게 유지
   - 검증 기준 갱신

2. **일관성 확인**
   - Phase 추가 시 기존 패턴 따르기
   - 범위 변경 시 "범위 외" 섹션 업데이트
   - 접근 방식 변경 시 "구현 전략" 섹션 업데이트
   - 자동/수동 검증 기준 구분 유지

3. **품질 기준 유지**
   - 새 내용에도 file:line 참조 포함
   - 측정 가능한 검증 기준
   - 명확하고 실행 가능한 언어

---

## Step 5: 변경 결과 제시

```
계획서를 업데이트했습니다: `.plans/[파일명]`

변경 사항:
- [구체적 변경 1]
- [구체적 변경 2]

개선된 점:
- [핵심 개선]
- [추가 개선]

추가 수정이 필요하면 알려주세요.
```

추가 피드백 반영 준비.

---

## 핵심 원칙

1. **회의적으로** — 문제 있어 보이는 변경 요청은 질문, 기술적 타당성 검증
2. **수술적으로** — 정밀 편집, 좋은 내용 보존, 필요한 것만 리서치
3. **철저하게** — 전체 계획서 읽고, 코드 패턴 리서치, 품질 기준 유지
4. **대화형으로** — 수정 전 확인, 방향 수정 허용
5. **미해결 질문 없이** — 변경이 질문을 일으키면 즉시 해결

---

## 검증 기준 가이드

수정 시 항상 두 카테고리 구분 유지:

### 자동 검증 (에이전트/CI가 실행 가능)
```
- [ ] 빌드 성공: `make build`
- [ ] 테스트 통과: `make test`
- [ ] 린트 통과: `make lint`
```

### 수동 검증 (사람이 확인 필요)
```
- [ ] UI에서 기능 정상 동작
- [ ] 성능이 허용 범위 내
- [ ] 엣지 케이스 처리 확인
```
