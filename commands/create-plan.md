# Implementation Plan (create-plan)

구현 계획을 조사 → 설계 → 단계별 계획서로 작성합니다.
humanlayer의 create_plan 워크플로우에서 영감을 받았으며, 범용적으로 사용 가능합니다.

## 실행 시작

### 파라미터가 있을 때
- 파일 경로가 주어지면 해당 파일을 **전체** 읽기
- 텍스트가 주어지면 요구사항으로 간주
- 즉시 Step 1로 진행

### 파라미터가 없을 때

```
구현 계획을 작성합니다. 다음을 알려주세요:

1. 구현할 기능 또는 작업 설명
2. 관련 컨텍스트, 제약조건
3. 참고할 기존 코드나 문서 (있다면)

팁: `/create-plan 기능 설명` 또는 `/create-plan path/to/ticket.md`로 직접 호출 가능
```

사용자 입력을 기다립니다.

---

## Step 1: 컨텍스트 수집 & 초기 분석

1. **언급된 파일 전체 읽기**
   - 티켓, 문서, 관련 코드 파일
   - Read 도구로 limit/offset 없이 전체 읽기
   - 서브에이전트에 위임하기 전에 메인 컨텍스트에서 먼저 읽기

2. **병렬 리서치 에이전트 실행** (최대 3개)
   - 관련 파일 위치 탐색
   - 기존 구현 패턴 분석
   - 유사 기능 또는 참고할 코드 검색

3. **리서치 결과 기반 파일 추가 읽기**
   - 에이전트가 발견한 관련 파일들을 메인 컨텍스트에서 읽기
   - 파일 전체를 읽어 완전한 이해 확보

4. **분석 결과 정리 & 질문**

```
조사 결과 정리:

**현재 상태:**
- [기존 구현 설명 + file:line 참조]
- [관련 패턴 또는 제약]

**확인이 필요한 사항:**
- [코드 조사로 해결 못한 기술적 질문]
- [비즈니스 로직 확인]
```

코드 조사로 답할 수 없는 질문만 사용자에게 묻습니다.

---

## Step 2: 리서치 & 설계 옵션

사용자 피드백 반영 후:

1. **수정사항이 있으면** 추가 리서치로 검증
2. **설계 옵션 제시**

```
**설계 옵션:**
1. [옵션 A] — [장점/단점]
2. [옵션 B] — [장점/단점]

**추천:** [옵션 X] — [이유]

어떤 접근 방식이 좋을까요?
```

---

## Step 3: 계획 구조 확인

접근 방식 합의 후:

```
계획 구조 제안:

## 개요
[1-2문장 요약]

## 구현 단계:
1. [Phase 이름] — [달성 목표]
2. [Phase 이름] — [달성 목표]
3. [Phase 이름] — [달성 목표]

이 구조가 적절한가요?
```

구조 승인 후 상세 계획 작성으로 진행합니다.

---

## Step 5: 리뷰 & 반복

1. **계획서 위치 안내**:
```
구현 계획 작성 완료:
`.plans/YYYY-MM-DD-description.md`

확인해 주세요:
- Phase 범위가 적절한가요?
- 검증 기준이 충분히 구체적인가요?
- 빠진 기술적 세부사항이 있나요?
- 누락된 엣지 케이스가 있나요?
```

2. **피드백 반영** — Phase 추가/제거, 기술 접근 조정, 범위 변경

---

## 핵심 원칙

1. **회의적으로** — 모호한 요구사항 질문, 잠재적 이슈 조기 식별, 가정하지 말고 코드로 확인
2. **대화형으로** — 한번에 전체 작성하지 않기, 각 단계에서 합의, 방향 수정 허용
3. **철저하게** — 컨텍스트 파일 전체 읽기, 병렬 리서치, 구체적 file:line 참조, 측정 가능한 검증 기준
4. **실용적으로** — 점진적/테스트 가능한 변경 집중, 마이그레이션/롤백 고려, "하지 않는 것" 명시
5. **미해결 질문 없이** — 계획서에 열린 질문이 남으면 즉시 조사하거나 확인 요청

## 일반 패턴

### DB 변경:
스키마/마이그레이션 → Store 메서드 → 비즈니스 로직 → API 노출 → 클라이언트 업데이트

### 새 기능:
기존 패턴 리서치 → 데이터 모델 → 백엔드 로직 → API → UI

### 리팩토링:
현재 동작 문서화 → 점진적 변경 계획 → 호환성 유지 → 마이그레이션 전략
