# Implement Plan (계획서 구현)

승인된 구현 계획서(`.plans/`)를 Phase별로 실행합니다.
각 Phase 완료 후 자동 검증 → 수동 확인을 거쳐 다음 Phase로 진행합니다.

---

## 실행 시작

### 계획서 경로가 주어졌을 때
- 계획서를 **전체** 읽기 (limit/offset 없이)
- 기존 체크마크 (`- [x]`) 확인 → 이미 완료된 Phase 파악
- 계획서에 언급된 원본 티켓, 참고 파일도 즉시 읽기
- Step 1로 진행

### 파라미터 없이 실행했을 때

1. `.plans/` 디렉토리에서 가장 최근 계획서 탐색
2. 파일이 없으면 사용자에게 경로 요청
3. 파일이 여러 개면 파일명 날짜 기준 **가장 최근** 선택

```
계획서를 구현합니다.

최근 계획서: .plans/2026-04-08-add-search-filter.md
이 계획서를 읽고 구현을 시작합니다.

팁: 직접 지정도 가능합니다:
/implement-plan .plans/2026-04-08-add-search-filter.md
```

---

## Step 1: 계획서 이해 & 컨텍스트 로딩

1. **계획서 전체 읽기**
   - 모든 Phase, 변경 사항, 검증 기준 파악
   - "범위 외 (하지 않는 것)" 확인

2. **병렬 에이전트로 현재 상태 확인**

   - **`codebase-locator`** — 계획서에 언급된 파일들이 현재 존재하는지, 위치 변경 없는지 확인
   - **`docs-locator`** — 관련 리서치나 이전 핸드오프가 있는지 탐색

3. **계획서에 언급된 핵심 파일 직접 읽기**
   - 에이전트에 위임하지 않고 메인 컨텍스트에서 읽기
   - 수정 대상 파일의 현재 상태 파악

4. **코드베이스 현실과 계획서 비교**
   - 계획서 작성 이후 변경된 부분이 있는지 확인
   - 불일치가 있으면 사용자에게 보고

---

## Step 2: 구현 진행 (Phase별)

각 Phase에 대해:

### 2a. Phase 시작 알림

```
Phase [N]: [이름] 시작합니다.

변경 대상:
- [파일 1]: [변경 요약]
- [파일 2]: [변경 요약]
```

### 2b. 코드 구현

- 계획서의 의도를 따르되, 현실에 맞게 적응
- 기존 코드 패턴과 컨벤션 준수
- 필요시 **`codebase-pattern-finder`** 에이전트로 유사 구현 참고
- Phase 내 각 항목 구현 후 계획서의 체크박스 업데이트 (`- [ ]` → `- [x]`)

### 2c. 불일치 발견 시

계획서와 현실이 맞지 않으면 **즉시 중지**하고 보고:

```
⚠️ Phase [N] 불일치 발견:

예상: [계획서에 적힌 내용]
실제: [현재 상태]
영향: [왜 이것이 문제인지]

어떻게 진행할까요?
1. 계획 수정 후 계속
2. 현재 상태에 맞게 적응
3. 중단
```

### 2d. 자동 검증

계획서의 "자동 검증" 항목 실행:

```bash
# 계획서에 명시된 검증 명령 실행
# 예시:
make build    # 빌드 확인
make test     # 테스트 통과
make lint     # 린트 통과
```

- 실패 시 원인 파악 후 수정
- 수정 후 다시 검증
- 모든 자동 검증 통과 후 다음 단계

### 2e. 수동 검증 요청

```
✅ Phase [N] 완료 — 수동 검증 필요

자동 검증 통과:
- [x] 빌드 성공
- [x] 테스트 통과
- [x] 린트 통과

수동 확인 부탁드립니다:
- [ ] [수동 검증 항목 1]
- [ ] [수동 검증 항목 2]

수동 테스트 완료 후 알려주시면 Phase [N+1]로 진행합니다.
```

**사용자가 여러 Phase 연속 실행을 요청한 경우**, 마지막 Phase에서만 수동 확인 일시정지.

## Step 3: 다음 Phase 또는 완료

### 다음 Phase가 있으면
- Step 2를 반복

### 모든 Phase 완료 시

```
🎉 모든 Phase 구현 완료!

완료된 Phase:
- Phase 1: [이름] ✅
- Phase 2: [이름] ✅
- Phase 3: [이름] ✅

다음 단계 추천:
- /validate-plan — 구현 결과 전체 검증
- /workcheck — 영향 분석 + 스모크 테스트
- /workfinish — 커밋 + PR 생성
```

---

## 이어서 구현하기 (Resume)

계획서에 기존 체크마크가 있으면:
- 완료된 작업은 신뢰 (이상해 보이면 검증)
- 첫 번째 미완료 항목(`- [ ]`)부터 재개
- 진행 상황을 계획서 체크박스에 실시간 반영

---

## 핵심 원칙

1. **계획서를 가이드로, 판단은 직접** — 계획의 의도를 따르되, 기계적으로 따르지 않음
2. **Phase별 완전 구현** — 다음 Phase 전에 현재 Phase 완전 완료
3. **불일치 즉시 보고** — 숨기거나 임의로 결정하지 않음
4. **체크박스 실시간 업데이트** — 계획서 파일을 진행 상황 추적에 활용
5. **자동 검증 후 수동 확인** — 두 단계 모두 통과해야 Phase 완료
