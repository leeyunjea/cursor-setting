# Validate Plan (구현 검증)

구현 계획서가 올바르게 실행되었는지 검증합니다.
모든 검증 기준을 확인하고 이탈이나 이슈를 식별합니다.

---

## 실행 시작

### 계획서 경로가 주어졌을 때
- 계획서를 **전체** 읽기
- Step 1로 진행

### 파라미터 없이 실행했을 때
1. `.plans/` 디렉토리에서 최근 계획서 탐색
2. 기존 대화 컨텍스트에서 구현 중인 계획서 파악
3. 가장 최근 계획서 선택

```
구현 결과를 검증합니다.

최근 계획서: .plans/2026-04-08-add-search-filter.md
이 계획서의 구현 결과를 확인합니다.

팁: /validate-plan .plans/파일명.md
```

---

## Step 1: 컨텍스트 수집

1. **계획서 전체 읽기**
   - 모든 Phase와 변경 사항 파악
   - 체크마크 상태 확인 (`- [x]` vs `- [ ]`)
   - 자동/수동 검증 기준 목록화

2. **구현 증거 수집** (병렬)

```bash
# 최근 커밋 확인
git log --oneline -20

# 구현 커밋들의 변경 범위
git diff HEAD~N..HEAD --stat   # N은 구현 커밋 수

# 전체 검증 실행
make build && make test && make lint
```

3. **병렬 에이전트로 구현 상태 확인**

   - **`codebase-locator`** — 계획서에 명시된 파일 변경이 실제로 존재하는지 확인
   - **`codebase-analyzer`** — 핵심 구현부가 계획서 의도대로 동작하는지 분석

---

## Step 2: 체계적 검증

각 Phase에 대해:

### 2a. 완료 상태 확인
- 계획서의 체크마크 확인
- 실제 코드가 체크된 내용과 일치하는지 검증

### 2b. 자동 검증 실행
- 계획서의 "자동 검증" 항목의 모든 명령 실행
- 성공/실패 기록
- 실패 시 원인 조사

### 2c. 수동 검증 항목 정리
- 사용자가 확인해야 할 항목 목록화
- 구체적 확인 단계 제시

### 2d. 코드 리뷰 관점 확인
- 에러 조건 처리 여부
- 누락된 유효성 검사
- 기존 기능 영향 가능성
- 테스트 커버리지

---

## Step 3: 검증 리포트 생성

```markdown
## 검증 리포트: [계획서 이름]

### 구현 상태
✅ Phase 1: [이름] — 완전 구현
✅ Phase 2: [이름] — 완전 구현
⚠️ Phase 3: [이름] — 부분 구현 (아래 이슈 참조)

### 자동 검증 결과
✅ 빌드 성공: `make build`
✅ 테스트 통과: `make test`
⚠️ 린트 경고: `make lint` (경고 3개)

### 코드 리뷰 소견

#### 계획 일치:
- [변경 사항]이 계획대로 구현됨
- [패턴]을 올바르게 따름

#### 계획 이탈:
- [file:line] — [이탈 내용] (개선/문제)
- [file:line] — [추가된 변경] (계획에 없었음)

#### 잠재적 이슈:
- [성능, 보안, 호환성 관련 우려]
- [누락된 에러 처리]

### 수동 테스트 필요:
1. UI 기능:
   - [ ] [확인 항목]
   - [ ] [에러 상태 테스트]

2. 통합:
   - [ ] [기존 기능과 호환 확인]
   - [ ] [성능 확인]

### 권장 사항:
- [머지 전 해결할 것]
- [추가 테스트 제안]
- [문서 업데이트 필요 사항]
```

## Step 4: 후속 조치 안내

```
검증 완료! 결과 요약:

✅ 통과: [N]개 항목
⚠️ 주의: [N]개 항목
❌ 실패: [N]개 항목

다음 단계 추천:
- [이슈가 있으면] 이슈 수정 후 다시 /validate-plan
- [모두 통과하면] /workcheck → /workfinish
```

---

## 핵심 원칙

1. **모든 자동 검증 실행** — 검증 명령 건너뛰지 않기
2. **증거 기반** — 추측이 아닌 실제 코드와 테스트 결과로 판단
3. **비판적 사고** — 구현이 진짜로 문제를 해결하는지 의문
4. **유지보수성 고려** — 장기적으로 관리 가능한지
5. **건설적** — 이슈 발견 시 해결책도 함께 제안

---

## 에이전트 활용

| 상황 | 에이전트 |
|------|---------|
| 파일 변경 존재 확인 | `codebase-locator` |
| 구현 의도 일치 분석 | `codebase-analyzer` |
| PR 리스크 사전 확인 | `pr-review-assistant` |
| 엔드포인트 변경 영향 | `endpoint-analysis` |

## 추천 워크플로우

```
/create-plan 기능 설명        ← 1. 계획 수립
/iterate-plan 피드백          ← 2. 계획 수정 (반복)
/implement-plan               ← 3. 계획 구현
/validate-plan                ← 4. 구현 검증     ← 지금 여기
/workcheck                    ← 5. 영향 + 스모크
/workfinish                   ← 6. 커밋 + PR
```
