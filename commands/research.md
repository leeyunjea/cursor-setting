# Research (코드베이스 리서치)

코드베이스를 체계적으로 탐색하여 리서치 문서를 생성합니다.
humanlayer의 research_codebase 워크플로우에서 영감을 받았습니다.

**핵심 원칙: 있는 그대로 문서화. 개선 제안이나 비평은 하지 않습니다.**

---

## 실행 시작

### 질문/주제가 주어졌을 때
- 언급된 파일이 있으면 **전체** 읽기
- 즉시 Step 1로 진행

### 파라미터 없이 실행할 때

```
코드베이스 리서치를 시작합니다.

리서치 질문 또는 관심 영역을 알려주세요.
병렬 에이전트로 관련 컴포넌트와 연결 관계를 분석합니다.

예시:
- "인증 플로우가 어떻게 동작하는지 알고 싶다"
- "결제 모듈의 구조를 파악하고 싶다"
- "이 프로젝트의 테스트 패턴을 이해하고 싶다"
```

---

## Step 1: 직접 언급된 파일 읽기

사용자가 특정 파일(티켓, 문서, JSON 등)을 언급하면:
- Read 도구로 limit/offset 없이 **전체** 읽기
- 서브에이전트에 위임하기 전에 메인 컨텍스트에서 먼저 읽기
- 이렇게 해야 리서치 분해 시 완전한 컨텍스트를 가질 수 있음

---

## Step 2: 리서치 질문 분해

1. 질문을 구성 가능한 리서치 영역으로 분해
2. 조사할 컴포넌트, 패턴, 개념 식별
3. 관련 디렉토리, 파일, 아키텍처 패턴 파악

---

## Step 3: 병렬 에이전트 실행

최대 4개의 에이전트를 동시 실행 (이름으로 명시 호출):

### 에이전트별 역할

**`codebase-locator`** — 파일 위치 탐색:
- "X 관련 파일이 어디에 있는지 찾아라"
- Glob, Grep으로 파일 패턴 검색
- 디렉토리 구조 매핑

**`codebase-analyzer`** — 구현 분석:
- "X가 어떻게 동작하는지 분석해라"
- 코드 흐름 추적, 데이터 플로우 파악
- 함수 호출 체인, 의존성 관계

**`codebase-pattern-finder`** — 패턴 탐색:
- "이 프로젝트에서 X 패턴의 사례를 찾아라"
- 유사 구현, 컨벤션 확인
- 테스트 패턴, 에러 처리 방식

**`docs-locator`** — 과거 문서 탐색:
- `.plans/`, `.research/`, `.handoffs/`에서 관련 과거 문서 검색
- 이전에 같은 주제를 조사한 적 있는지 확인
- 관련 핸드오프에서 Learnings 활용

### 에이전트 지침
- 각 에이전트는 **문서화 전용** — 평가/비평 금지
- 구체적 file:line 참조 반환
- 발견한 예시와 사용 패턴을 있는 그대로 기술
- 과거 문서가 발견되면 **`docs-analyzer`**로 인사이트 추출 가능

---

## Step 4: 결과 종합

**모든 에이전트 완료를 기다린 후** 종합:

1. 코드베이스 조사 결과를 우선 (1차 소스)
2. 컴포넌트 간 연결 관계 정리
3. 구체적 파일 경로와 라인 번호 포함
4. 아키텍처 패턴과 설계 결정사항 기록

---

## Step 5: 리서치 문서 작성

`.research/YYYY-MM-DD-description.md`에 작성합니다.

### 리서치 문서 템플릿

```markdown
---
date: [ISO 형식 날짜+시간]
git_commit: [현재 커밋 해시]
branch: [현재 브랜치]
project: [프로젝트 이름]
topic: "[리서치 주제]"
tags: [research, 관련 컴포넌트명]
---

# Research: [리서치 주제]

## 리서치 질문
[원래 사용자 질문]

## 요약
[핵심 발견사항으로 질문에 답변, 존재하는 것 기술]

## 상세 분석

### [컴포넌트/영역 1]
- 존재하는 것 설명 (file:line 참조)
- 다른 컴포넌트와의 연결
- 현재 구현 상세

### [컴포넌트/영역 2]
...

## 코드 참조
- `path/to/file.ext:123` — 무엇이 있는지 설명
- `another/file.ext:45-67` — 코드 블록 설명

## 아키텍처 문서
[코드베이스에서 발견된 패턴, 컨벤션, 설계 구현]

## 미해결 질문
[추가 조사가 필요한 영역]
```

---

## Step 6: 결과 요약 & 후속 질문

1. 사용자에게 간결한 요약 제시
2. 핵심 파일 참조 포함
3. 후속 질문이 있으면 같은 리서치 문서에 추가

```
리서치 결과를 정리했습니다:
`.research/YYYY-MM-DD-description.md`

핵심 발견:
- [발견 1 요약]
- [발견 2 요약]

추가 궁금한 점이 있으면 알려주세요. 같은 문서에 이어서 추가합니다.
```

---

## 핵심 원칙

1. **있는 그대로 문서화** — 개선 제안, 비평, 루트 원인 분석 금지 (요청하지 않는 한)
2. **파일 전체 읽기** — limit/offset 없이 Read
3. **병렬 실행** — 서로 다른 영역을 동시 조사
4. **모든 에이전트 완료 대기** — 종합하기 전에 모든 결과 확보
5. **구체적 참조** — 항상 file:line 형식으로 코드 위치 명시
6. **메인 컨텍스트 우선** — 핵심 파일은 서브에이전트 전에 직접 읽기
